# Tableau Workbook Analyzer - Git / GitHub 運用ルール

本プロジェクトにおける、Gitのコミット、ブランチ管理、およびPull Request（PR）のルールを定義します。開発時は常にこのルールを遵守してください。

## 1. ブランチ戦略 (Feature Branch Workflow)
*   **`main` ブランチ**: 常にデプロイ可能で安定した状態を保ちます。直接のコミット (`git push origin main`) は原則禁止です。
*   **作業ブランチ**: 開発を行う際は、必ず `main` から新しいブランチを切って作業します。
    *   新機能: `feature/xxx` (例: `feature/twbx-parser`)
    *   バグ修正: `fix/xxx`
    *   テスト追加: `test/xxx`
    *   リファクタリング: `refactor/xxx`

## 2. コミットメッセージ規約 (Conventional Commits)
コミットメッセージは以下のプレフィックスから始め、何を行った変更かが一目でわかるようにします。

*   `feat:` 新機能の追加
*   `fix:` バグ修正
*   `test:` テストコードの追加・修正（TDDのフェーズ等）
*   `docs:` ドキュメントの更新
*   `refactor:` リファクタリング（機能やバグ修正を含まないコードの改善）
*   `chore:` ビルド設定、パッケージマネージャーの設定などの雑務
*   `style:` コードのフォーマット変更（空白、セミコロンなど、機能に影響しない変更）

*(例) `feat: twbxファイルのドラッグ＆ドロップ読み込み機能を追加`*

## 3. Pull Request (PR) の作法
1.  **PRの作成**: 作業ブランチでの開発が完了したら、`main` に向けて PR を作成します。
2.  **テンプレートの使用**: プロジェクトに設定されている PR テンプレートに従い、「変更内容」「関連するIssue（あれば）」「テスト方法」を明記します。
3.  **セルフレビュー**: PR を提出する前に、必ずローカルで `npm run test` (テスト) と `npm run lint` (静的解析) が通ることを確認します。
4.  **マージの条件**:
    *   CI（GitHub Actions）の全てのチェック（Lint, TypeCheck, Test）が **Pass (緑色)** していること。
    *   （チーム開発の場合）最低1名の Approve（承認）を得ること。
